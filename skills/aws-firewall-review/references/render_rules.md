# Report rendering rules

These rules govern how to turn the evidence JSON (from the
`aws-firewall-review` MCP server's `collect_security_groups` tool) plus the
template (`assets/report_template.md`) into a finished report. The tool returns
`{"accounts": [...]}`; render one report per account object, each of which has
the `metadata` / `global_summary` / `regions` shape below.

## Template markers

The template uses three marker conventions:

- `HINT:` lines are instructions to you. Read, follow, and **strip them from the output**. A HINT may span several lines (continuation lines indented beneath it) — strip all of them.
- `LOOP: ... ENDLOOP` blocks repeat once per item in the corresponding evidence collection. Strip the marker lines themselves. If the collection is empty, emit nothing for the block (unless a HINT immediately above specifies fallback text, e.g. "If zero HIGH findings, write '...'").
- `<<PLACEHOLDER>>` tokens are substituted with values derived from the evidence. Never leave a placeholder in the final output. If a value genuinely cannot be derived, write `[unknown]`.

## Evidence shape

```
metadata.aws_account_id, metadata.generated_at, metadata.regions_scanned, metadata.aws_caller_arn
global_summary.total_security_groups, .unused_security_groups
global_summary.findings_by_severity {HIGH, MEDIUM, LOW}
global_summary.internet_exposed_sensitive_services[]
regions[].region
regions[].error            (present only when the SCP denied access — look for "UnauthorizedOperation")
regions[].security_groups[] each with:
    group_id, group_name, description, vpc_id, region, tags{}, is_default,
    in_use, attached_eni_count, attached_enis[],
    ingress_rules[], egress_rules[],
    risk_summary { total_findings, high, medium, low, findings[] }
```

Each rule in `ingress_rules[]` / `egress_rules[]` has: `direction`, `protocol`, `port_range`, `from_port`, `to_port`, `sources[]` (each `{type, value, description}` where type is cidr | cidr_v6 | security_group | prefix_list), and `risk_findings[]` (each `{severity, category, detail}`).

**The `risk_findings[]` are authoritative.** They were generated by deterministic logic in the collector. Do not invent findings beyond them; you may group, summarise, and prioritise.

## Values pulled directly from evidence

- Review date ← `metadata.generated_at` (format as `DD MMM YYYY`)
- AWS account ID ← `metadata.aws_account_id`
- Severity counts ← `global_summary.findings_by_severity`
- Distinct-SGs-affected per severity ← count of SGs where `risk_summary.<severity> > 0`
- Total SG count ← `global_summary.total_security_groups`

## Region counts — always compute from the data, never copy the example

Region scope numbers are a common source of error. Derive every count by
inspecting `regions[]` directly; do not reuse the numbers in
`references/example_report.md` (they describe a different run).

- **Total regions** = length of `metadata.regions_scanned`.
- **Denied (out-of-scope) regions** = count of `regions[]` where `error` contains `UnauthorizedOperation`.
- **Accessible regions** = total regions − denied regions. This INCLUDES accessible regions that returned zero security groups, so do not undercount by dropping the empty ones.
- **Regions with workloads** = accessible regions whose `security_groups[]` is non-empty.

Sanity check before writing the scope sentence: denied + accessible must equal
total. If the example report and the data disagree, the data wins.

## Defaults for fields not in evidence

- Version: `1.0`
- Preparer: `Security Engineering, DiscoveryLoft Vietnam` (or the user's name if they ask)
- Approver: leave as `[Name], [Role]` for manual fill-in
- Quarter tag/label: derive from `metadata.generated_at` (e.g. `2026Q2`, "Q2 2026")
- Next review: 90 days after `metadata.generated_at`
- Report sequence (DLVN-SEC-REV-NNN): ask the user, or default to `002`

## Finding IDs — must be stable across re-runs

Sort findings: HIGH first, then MEDIUM, then LOW. Within a severity, sort by region (alphabetical), then `group_id`, then ingress rule index. Number sequentially `F-001`, `F-002`, … The cross-reference form is `DLVN-FW-<QUARTER_TAG>-F001`.

## Default owners and due dates

- SG name starts with `sgr-mgmt-` / `devsecops` / `wazuh` / `vpn` → owner `DevSecOps — Security Engineer`
- SG name contains `eks` / `alb` / `lb` → owner `DevOps Lead`
- LOW findings → owner `Security Eng.`
- Due dates measured from review date: HIGH +7 days, MEDIUM +30 days (use +14 days when the exposure is an internal observability or data-store service reachable from the internet — Loki, Prometheus, Elasticsearch, Mongo, Redis, etc.), LOW = end of next quarter
- Observations (section 4) → due date = next review date

## Assessment rating

- `EFFECTIVE` — zero HIGH findings AND 3 or fewer MEDIUM
- `PARTIALLY EFFECTIVE` — any HIGH findings OR more than 3 MEDIUM
- `NOT EFFECTIVE` — 2 or more HIGH findings AND no remediation plan documented

Justify the rating in one sentence naming the deciding factor.

## Section 4 — Additional Observations: patterns to surface

Emit one row per pattern present (these are hygiene items, NOT formal findings):

| Pattern | Detection |
|---|---|
| Orphaned SG | `attached_eni_count == 0` AND the SG appears in another SG's `ingress_rules[].sources[]` with `type: security_group` |
| Stale temporary rule | any rule `description` containing `tmp`, `temp`, `manual`, `mannuly`, `test` |
| Default VPC SG in use | `is_default == true` AND `attached_eni_count > 0` (default SGs *not* in use are normal — no row needed) |
| Untagged SG | `tags == {}` and not a default SG |
| Cross-cloud migration residue | SG name/description mentions `gcp`, `azure`, `on-prem`, `migration` |

If no patterns apply, omit section 4 entirely (heading and table).

## Output style rules

1. Output only the rendered markdown — no preamble, no postamble, no commentary, no leftover markers or placeholders.
2. No emojis.
3. No external compliance framework references (SOC 2, ISO 27001, NIST, PCI-DSS) unless the user explicitly asks. The driver is DLVN-SEC-POL-002.
4. Date format `DD MMM YYYY`; short form in table cells `DD Mmm` (e.g. `28 Apr`).
5. Severity labels exactly `HIGH`, `MEDIUM`, `LOW`, `Info`. Status labels exactly `Open`, `In Progress`, `Resolved`, `Risk Accepted`. New findings default to `Open`.
6. In Appendix B, bold the H count when greater than zero (e.g. `**1**/2/1`).
7. Code-format `sg-…`, `vpc-…`, `arn:…`, ports, and CIDRs in backticks.
8. British English spelling (`organisation`, `authorise`, `behaviour`, `centre`).

## Self-check before delivering

- No line begins with `HINT:`, `LOOP:`, or `ENDLOOP`.
- No `<<...>>` placeholder remains.
- Every finding row has an Owner and a Due Date.
- Severity counts in section 2.1 match the number of finding rows in 3.1 / 3.2 / 3.3.
- Appendix B row count equals the total SG count.
- No emojis; no external framework names (unless requested).
- Output begins with `# AWS Firewall Configuration Review` and ends with the last Appendix C bullet.
